## 一、防破解体系

### 1.1 分层防护架构

```
┌─────────────────────────────────────────────┐
│  第 4 层：服务端兜底                          │
│  核心算法放后端 API，客户端仅持有调用 token    │
├─────────────────────────────────────────────┤
│  第 3 层：许可证验证                          │
│  HWID 绑定 + HMAC-SHA256 签名 + 过期控制      │
│  + 服务端二次验证（联网时）                    │
├─────────────────────────────────────────────┤
│  第 2 层：运行时防护                          │
│  文件完整性校验 + 反调试检测 + self-defending  │
├─────────────────────────────────────────────┤
│  第 1 层：编译/打包层                          │
│  代码混淆 + V8 字节码 + ASAR 防解压            │
└─────────────────────────────────────────────┘
```

### 1.2 第 1 层：编译/打包层防护

**目标**：让破解者无法直接阅读源代码。

| 手段 | 工具 | 效果 |
|------|------|------|
| JavaScript 混淆 | `javascript-obfuscator` | 变量名/字符串/控制流混淆，增加逆向成本 |
| V8 字节码编译 | `bytenode` | 主进程核心逻辑编译为 .jsc，无法直接读源码 |
| ASAR 防解压 | `asarmor` | 修改偏移值 + 注入假文件，阻止 asar 工具解包 |
| 字符串加密 | 自定义 StringLoader | 密钥/API endpoint 等敏感字符串加密存储 |

**关键决策**：主进程 License 校验代码必须进 V8 字节码，不保留 JavaScript 明文。

### 1.3 第 2 层：运行时防护

**文件完整性校验**：启动时计算所有 .asar/.jsc 文件的 hash，与发行版清单比对。任一文件被篡改 → 拒绝启动。

**反调试检测**：
- 检测 DevTools 是否被打开（Electron 生产模式默认关闭，但可通过命令行 `--inspect` 重新开启）
- 使用 `debugProtection: true, debugProtectionInterval: 2000` 自动关闭 DevTools
- 检测 Node.js `--inspect` 标志位

**Self-Defending 代码**：混淆器开启 self-defending 模式，任何对混淆代码的结构性修改都会触发执行中断。

### 1.4 第 3 层：许可证验证

**离线验证（核心机制）**：

```
License Token = Base64(JSON Payload) + "." + HMAC-SHA256(JSON Payload, SecretKey)
```

Payload 结构：
```json
{
  "user_id": "uuid",
  "hwid": "CPU序列号+硬盘序列号+网卡MAC 的 sha256",
  "plan": "basic|pro|enterprise",
  "issued_at": "2026-07-02T00:00:00Z",
  "expires_at": "2027-07-02T00:00:00Z"
}
```

**在线补充验证**（联网时）：服务端记录 License 使用设备数，同一 License 超过 N 台设备 → 触发风控。

### 1.5 第 4 层：服务端兜底

以下逻辑**永远不放在客户端**：
- License 生成和签发（服务端独立服务）
- 论文检索核心算法（API 调用）
- AI 写作 Prompt 模板（服务端下发，客户端不可见）
- 任何涉及商业机密的处理逻辑

客户端只是瘦壳 + 渲染层，关键能力通过 API 调用。破解客户端的收益被限制在 UI 层面，拿不到核心算法。

## 三、外部算力模型配置工具（Model Gateway）

### 3.1 问题定义

当前 SPEC.md 的设计隐式耦合了单一 AI 提供商。但实际场景中：
- C 端用户可能用自备 API Key（DeepSeek / Qwen / OpenAI）
- B 端客户要求私有化部署 Ollama / vLLM
- 学术用户可能需要切换到 arXiv 专用模型
- 我们自己也需要随时切换底层模型以控制成本

需要一个**模型网关**，让用户（和管理员）可以动态配置、切换、测试多个 AI 模型提供商。

### 3.2 架构设计

```
┌──────────────────────────────────────────────┐
│              前端配置面板                      │
│  ┌─────────────────────────────────────┐     │
│  │ 模型列表 │ 提供商 │ 测试连接 │ 切换 │     │
│  └─────────────────────────────────────┘     │
├──────────────────────────────────────────────┤
│          Backend Model Gateway               │
│  ┌──────────────────────────────────────┐    │
│  │  统一 OpenAI 兼容层                   │    │
│  │  provider 路由 │ 负载均衡 │ fallback │    │
│  └──────────────────────────────────────┘    │
├──────────────────────────────────────────────┤
│  Provider 适配器                             │
│  ┌───────┐ ┌───────┐ ┌──────┐ ┌──────┐      │

```

### 3.3 新增数据表（追加到 SPEC.md 数据库模型）

```sql
-- 模型提供商
CREATE TABLE model_providers (
    id UUID PRIMARY KEY DEFAULT gen_uuid(),
    name VARCHAR(50) NOT NULL UNIQUE,         -- openai / deepseek / qwen / ollama / custom
    display_name VARCHAR(100) NOT NULL,        -- OpenAI / DeepSeek / 通义千问
    base_url VARCHAR(500) NOT NULL,            -- https://api.openai.com/v1
    api_type VARCHAR(20) DEFAULT 'openai',     -- openai / anthropic / custom
    is_builtin BOOLEAN DEFAULT false,          -- 系统预置不可删除
    is_enabled BOOLEAN DEFAULT true,
    created_at TIMESTAMP DEFAULT now()
);

-- 用户模型配置（个人 API Key 或选择系统模型）
CREATE TABLE user_model_configs (
    id UUID PRIMARY KEY DEFAULT gen_uuid(),
    user_id UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider_id UUID NOT NULL REFERENCES model_providers(id),
    api_key_encrypted TEXT,                    -- 用户自备 Key（加密存储）
    model_name VARCHAR(100) NOT NULL,          -- gpt-4o / deepseek-chat
    is_default BOOLEAN DEFAULT false,          -- 是否设为默认
    priority INT DEFAULT 0,                    -- 优先级（多模型 fallback 时使用）
    created_at TIMESTAMP DEFAULT now(),
    UNIQUE(user_id, provider_id, model_name)
);
```

### 3.5 统一调用层（Gateway 核心逻辑）

```python
# 伪代码，实际实现放在 backend/app/services/model_gateway.py
class ModelGateway:
    """统一模型调用网关"""

    async def chat(
        self,
        user_id: UUID,
        messages: list[dict],
        provider_id: UUID | None = None,
    ) -> ChatResponse:
        # 1. 确定使用哪个 provider
        config = await self._get_user_config(user_id, provider_id)

        # 2. 解析实际 API endpoint 和 key
        provider = await self._get_provider(config.provider_id)
        api_key = self._decrypt(config.api_key_encrypted) or provider.default_key

        # 3. 转换为 OpenAI 兼容格式，调用
        response = await self._call_openai_compatible(
            base_url=provider.base_url,
            api_key=api_key,
            model=config.model_name,
            messages=messages,
        )

        # 4. 记录调用日志（用于计费和监控）
        await self._log_usage(user_id, provider.id, config.model_name, response.usage)

        return response
```
