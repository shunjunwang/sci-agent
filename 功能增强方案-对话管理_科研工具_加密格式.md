## 一、对话搜索

### 1.1 竞品参考

| 产品 | 搜索方式 | 搜索粒度 |
|------|----------|----------|
| **ChatGPT** | 侧边栏顶部搜索框 | 标题匹配（不含正文内容搜索），输入关键词后实时过滤 |
| **Perplexity** | Library 内全局搜索 + Threads/Collections 组织 | 标题+内容全文搜索，支持线程归档和按集合分组 |
| **Claude** | 侧边栏搜索、项目内搜索 | 标题+对话内容全量搜索（内置向量语义搜索） |
| **Notion AI** | 数据库全文检索 | 标题+正文+注释，支持按标签/属性筛选 |
| **Cursor** | Command Palette 搜索 | 项目级代码+对话混合搜索 |

### 1.2 我方方案

基于 UI_LAYOUT.md 现有的左侧栏结构，在会话历史列表上方增加搜索入口。

**三级搜索模式：**

```
┌────────────────────────┐
│  🔍 搜索对话...         │  ← 输入框，支持快捷键 Ctrl+K / Cmd+K
│  ┌──────────────────┐  │
│  │ 搜索范围：        │  │
│  │ ◉ 标题  ○ 问题   │  │  ← 初始模式：仅标题匹配（轻量、即时）
│  │ ○ 全局           │  │  ← 全局模式：标题+正文全文（需要点切换或长按）
│  └──────────────────┘  │
└────────────────────────┘
```

| 搜索模式 | 技术方案 | 响应速度 | 说明 |
|----------|----------|----------|------|
| **标题搜索** | SQL LIKE + pg_trgm 模糊匹配 | <50ms | 默认模式，输入即搜 |
| **问题搜索** | 搜索用户首条提问内容 | <100ms | 关联对话的开场问题 |
| **全局搜索** | pgvector 语义搜索 + FTS5 全文索引 | <500ms | 需主动切换，搜索对话正文内容 |

**技术细节：**

- 标题搜索：在 conversations 表的 title 字段建 GIN 索引（`CREATE INDEX idx_conversations_title_trgm ON conversations USING gin(title gin_trgm_ops);`），支持拼音首字母和模糊匹配
- 全局搜索：对每轮对话的 messages.content 做 embedding（复用已有 pgvector 基础设施），结合 PostgreSQL 全文搜索（`to_tsvector('simple', content)`），两级召回后 RRF 融合排序
- 搜索结果显示匹配片段高亮，点击直接跳转到对应对话位置

### 1.3 数据库改动

```sql
-- 会话表增加全文搜索向量列
ALTER TABLE conversations ADD COLUMN content_vector vector(1536);

-- 消息内容全文索引
CREATE INDEX idx_messages_content_fts ON messages USING gin(to_tsvector('simple', content));

-- 标题模糊搜索索引
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX idx_conversations_title_trgm ON conversations USING gin(title gin_trgm_ops);
```

### 2.3 API 设计

```
PATCH  /api/conversations/{id}/rename     { "title": "新标题" }
PATCH  /api/conversations/{id}/pin        { "pinned": true/false }
DELETE /api/conversations/{id}            # 软删除
POST   /api/conversations/{id}/restore    # 从回收站恢复
GET    /api/conversations/trash           # 回收站列表
DELETE /api/conversations/{id}/permanent  # 永久删除（回收站内操作）
```

## 四、加密格式方案

### 4.1 CEBX 分析与评估

**CEBX（Common e-Document of Blending XML）** 是方正阿帕比的版式文档标准，2013年被定为中国新闻出版行业标准。核心特性：

| 特性 | CEBX | 我方适用性 |
|------|------|------------|
| 版式固定 | 原版原式呈现，不可更改 | ✓ 符合需求 |
| DRM 加密 | Apabi DRM 体系，支持密级管理 | ✓ 符合需求 |
| 全文检索 | 支持 | ✓ |
| 脚本交互 | 支持动态脚本 | — 非必需 |
| 跨平台 | Windows / Linux / macOS / 移动端 | 需验证 |
| **致命缺陷** | 方正私有格式，生态封闭，浏览器不支持原生查看 | ✗ |

**结论**：CEBX 理念正确但生态过于封闭。不建议直接使用 CEBX，而是借鉴其设计思路，用开放技术栈自研等效方案。

### 4.2 推荐方案：三层加密文档体系

```
┌──────────────────────────────────────────────────────┐
│                    用户导出文档                        │
│                                                      │
│  ┌──────────┐    ┌──────────────┐    ┌────────────┐ │
│  │ 常规格式  │    │  加密格式     │    │  分享模式   │ │
│  │ .pdf     │    │  .scix       │    │ 临时链接    │ │
│  │ .docx    │    │  (自研封装)   │    │ + 水印      │ │
│  │ .md      │    │              │    │ + 阅后即焚  │ │
│  └──────────┘    └──────────────┘    └────────────┘ │
└──────────────────────────────────────────────────────┘
```

#### 4.2.1 .scix 加密格式设计

**格式结构：**

```
.scix 文件 = ZIP 容器
├── manifest.json       # 元数据：标题、作者、时间戳、签名
├── content.enc         # AES-256-GCM 加密的正文（HTML/Markdown）
├── assets/             # 加密的图片/附件
│   ├── fig1.png.enc
│   └── table1.csv.enc
└── signature.sig       # Ed25519 数字签名，防篡改
```

**技术栈：**

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 内容格式 | HTML5 / Markdown → 渲染为固定版式 | 保证原版原式，数学公式/图表无失真 |
| 加密算法 | AES-256-GCM | 军事级加密，每次加密使用随机 IV |
| 密钥派生 | PBKDF2-SHA256（用户密码 + salt） | 密钥不存储在服务器 |
| 数字签名 | Ed25519 | 防篡改 + 来源认证 |
| 容器格式 | ZIP (deflate) | 通用、压缩、可流式读取 |

**预览/查看方案：**

.scix 文件只能在 Sci-Agent 客户端内打开。打开流程：

```
用户双击 .scix → 客户端弹出密码输入框 → PBKDF2 派生密钥
→ 解密 content.enc → 签名验证 → 渲染为固定版式 HTML 视图
```

**防复制措施：**

| 措施 | 实现方式 | 防护层级 |
|------|----------|----------|
| 文本禁止选中 | CSS `user-select: none` + JS 拦截 | 基础 |
| 禁止右键菜单 | `contextmenu` 事件拦截 | 基础 |
| 禁止截图通知 | 检测 `Screen Capture API` / 打印遮挡 | 中等 |
| 动态水印 | 当前用户 ID + 时间戳，半透明平铺 | 中等 |
| 客户端渲染 | 正文不在 DOM 中明文出现，分段解密渲染 | 较高 |
| 打印限制 | 打印时叠加高密度水印，降低可用性 | 较高 |

> **诚实声明**：没有 100% 的防复制方案。面对手机拍照、虚拟机截图、OCR 等手段，任何 DRM 都存在绕过路径。本方案的商业价值在于：**大幅提高复制成本**，配合产品体验优势，让用户选择"留下"而非"搬走"。这正是 CEBX 的商业逻辑——不是技术无法破解，是破解的性价比不足以抵消迁移成本。

#### 4.2.2 分享模式（轻量 DRM）

针对"需要发给别人看但不想被复制"的场景：

```
创建分享链接
├── 有效期：1小时 / 24小时 / 7天 / 永久
├── 访问次数限制：1次 / 5次 / 无限
├── 水印：强制显示接收方信息 + 时间
├── 操作限制：
│   ├── ☑ 禁止复制文本
│   ├── ☑ 禁止下载
│   ├── ☐ 允许打印（加水印）
│   └── ☐ 允许评论
└── 额外验证：短信验证码 / 邮箱验证码
```

**技术实现：**

- 分享链接生成临时 Token，后端验证权限
- 内容通过 Web 加密通道投递，浏览器端渲染（类似 Google Docs 受保护的查看模式）
- 参考：VeryPDF DRM Protector 的权限控制模型 + SecureVault 的分享链接机制

### 4.3 竞品加密方案对比

| 方案 | 加密方式 | 防复制 | 迁移成本 | 我方适配 |
|------|----------|--------|----------|----------|
| 方正 CEBX | 私有格式 + DRM | ★★★★ | 极高（格式锁定） | 不采纳，生态封闭 |
| VeryPDF DRM | PDF + AES-256 + 权限控制 | ★★★★ | 中（PDF 生态） | 可参考权限模型 |
| SecureVault | 自建加密容器 + AES-256-GCM | ★★★ | 高（自研格式） | **核心参考** |
| Google Docs 受限模式 | 浏览器端渲染 + 禁止下载 | ★★ | 低（在线依赖） | 参考分享模式 |
| 纯水印 | 无加密 | ★ | 极低 | 仅作为辅助手段 |

## 六、风险与注意事项

| 风险项 | 说明 | 缓解措施 |
|--------|------|----------|
| **pgvector 性能** | 全局搜索随数据量增长可能变慢 | 限制搜索 scope（最近 N 条消息），加缓存层 |
| **加密密钥安全** | PBKDF2 派生密钥依赖用户密码强度 | 强制密码最低长度 8 位 + 密码强度指示器 |
| **DRM 过度预期** | 用户可能误以为加密=绝对安全 | UI 中明确说明防护边界，不夸大宣传 |
| **Mermaid 复杂图** | 超大流程图渲染性能 | 限制节点数，超过阈值降级为静态图片 |
| **工具广场膨胀** | Skill 包数量增加导致管理混乱 | 分类体系 + 评分排序 + 搜索过滤 |
